## 問題文

**問6** クーポン発行サービスに関する次の記述を読んで、設問1〜4に答えよ。

K社は、インターネット上でホテル、旅館及びレストラン（以下、施設という）の予約を取り扱う施設予約サービスを運営している。各施設は複数のプランを提供しており、利用者はその中から好みのプランを選んで予約する。会員向けサービスの拡充施策として、現在稼働している施設予約サービスに加え、クーポン発行サービスを開始することにした。

発行するクーポンは割引金額が大きく、クーポンの発行直後にトラフィックが集中することが想定される。発行に限数到達後にクーポン獲得処理が動作する場合の考慮も必要である。K社は、トラフィック集中時のリソース競合による応答速度の悪化を懸念して、ロック解放待ちを発生させない連番管理方式（以下、ロックなし方式という）の SQL 文（図4）を選定した。この SQL 文では、ロックを掛けずに参照し、主キー制約によってクーポン発行連番の重複レコード作成を防止する。

ここで、関数 COALESCE(A, B)は、A が NULL でないときは A、A が NULL のときは B を返す。"クーポンコード"、"会員コード" は、当該の値を格納する埋込み変数である。

### 〔クーポン発行サービスと施設予約サービスの E-R図〕

クーポン発行サービスと施設予約サービスで共用するデータベース（以下、予約サイトデータベースという）の E-R図（抜粋）を図1に示す。予約サイトデータベースは、E-R図のエンティティ名をテーブル名に、属性名を列名にして、適切なデータ型で表定義した関係データベースによってデータを管理する。

クーポン管理テーブルの列名の先頭に "獲得制限" が付く列は、クーポンの獲得制限、及びクーポンを利用した予約制限のチェック処理で使用し、チェックが必要な場合は NULL を設定する。"獲得制限_1枚取り" には、"同一会員1枚取りの獲得制限" のチェックが必要なときは 'Y' を、不要なときは NULL を設定する。

### 図2 "同一会員1枚取りの獲得制限" を制約とするためのSQL文

![図2 UNIQUE制約SQL](../images/2022春_問06_図02_UNIQUE制約SQL.png)

```sql
ALTER TABLE クーポン明細 ADD CONSTRAINT クーポン明細_IK1
UNIQUE(クーポンコード, 獲得制限_1枚取り, `[　d　]`)
```

L主任は、①予約テーブルの "クーポンコード"、"クーポン発行連番" に対して、UNIQUE 制約を付けた。

予約サイトデータベースでは、更新目的の参照処理と更新処理においてレコード単位にロックを掛け、多重処理を方言設定としている。ロックが掛かるとトランザクションが終了するまで他のトランザクションは同一レコードに対する処理はロック解除待ちとなる。

### 図4 ロックなし方式のSQL文

![図4 ロックなしSQL](../images/2022春_問06_図04_ロックなしSQL.png)

```sql
INSERT INTO クーポン明細(クーポンコード, クーポン発行連番, 獲得制限_会員コード, 獲得制限_1枚取り)
  (SELECT 発行済枚数+1, :会員コード AS (COALESCE(MAX( `[　e　]` ), 0) + 1)
   FROM クーポン明細 WHERE クーポンコード = :クーポンコード)
```

---

ロックなし方式をレビューした M 課長は、トラフィック集中時に主キー制約違反が発生することによって会員による再オペレーションが頻発するデメリットを指摘し、ロック解放待ちを発生させることによって更新が順番に行われる連番管理方式（以下、ロックあり方式という）の SQL 文も検討するように依頼した。

L主任は、クーポン管理テーブルに対して初期値が0の "発行済枚数" という列を追加し、このデータ項目のカウントアップによって連番管理をするロックあり方式の SQL 文（図5）を考案した。

---

### 図5 ロックあり方式のSQL文

![図5 ロックありSQL](../images/2022春_問06_図05_ロックありSQL.png)

```sql
UPDATE クーポン管理
WHERE クーポンコード = :クーポンコード AND 発行済枚数 < `[　f　]`
SET `[　g　]`
  SELECT :クーポンコード, 発行済枚数, :会員コード, 獲得制限_1枚取り
  FROM クーポン管理 WHERE クーポンコード = :クーポンコード
```

---

③**ロックあり方式では、図3のCRUD図の一部に変更が発生する**。

L 主任は、ロックなし方式とロックあり方式の比較を表3にまとめ、高負荷試験を実施した。

### 表3 ロックなし方式とロックあり方式の比較

![表3 ロック方式比較](../images/2022春_問06_表03_ロック方式比較.png)

> | 方式 | ロック解放待ちの発生 | 発生上限枚数到達後の再オペレーション |
> |------|-----------------|--------------------------------|
> | ロックなし | 発生しない | 副問合せで取得する発行済枚数がNULLになり、クーポン発行連番が最大値からやり直しになる。一クーポン明細テーブルの行に主キー制約違反があり再オペレーションが必要 |
> | ロックあり | 発生する | 変更が加わり、クーポン管理テーブルの行をロックしてから追加するようになり、主キー制約違反なく行を追加する |

高負荷試験の結果、どちらの方式でも最大トラフィック発生時のレスポンス・スループットが規定値以内に収まることが確認できた。そこで、会員による再オペレーションのデメリットを鑑みて、**ロックあり方式**を採用した。

## 解答と解説

### 設問1

**(1) E-R図の関連：a = 施設コード、b = プランコード、c = ↑（1対1）**

- **a = 施設コード**：E-R図でクーポン管理→施設への参照キー。クーポンは施設に属する。
- **b = プランコード**：クーポンが参照するプランを示す外部キー。
- **c = ↑（1対1の関係記号）**：クーポン明細と予約の関係。1枚のクーポンは一つの予約にのみ利用できる。

**IPA公式：a=施設コード、b=プランコード、c=↑（順不同）**

**(2) 正解：d = ALTER TABLE**

図2のSQL文は既存テーブルにUNIQUE制約を追加するもの。`ALTER TABLE テーブル名 ADD CONSTRAINT 制約名 UNIQUE(列名...)` の構文。

**IPA公式：d = ALTER TABLE**

**(3) 正解：1枚のクーポンは一つの予約だけに利用できる。（23字）**

予約テーブルの「クーポンコード」「クーポン発行連番」にUNIQUE制約を付けることで、同じクーポン（同じコードと連番の組み合わせ）が複数の予約に使われることを防ぐ。1クーポン＝1予約のみ使用可能という業務ルールを実現する。

---

### 設問2 正解：e = クーポン発行連番

ロックなし方式のSQLは、既存の最大発行連番に1を加算した値を新しい連番として INSERT する。  
`COALESCE(MAX(クーポン発行連番), 0) + 1` で、レコードがない場合（NULL）は0とし、1番から始まる連番を生成する。

**IPA公式：e = クーポン発行連番**

---

### 設問3

図5は `UPDATE クーポン管理 [ f ] WHERE クーポンコード = :クーポンコード AND 発行済枚数 < [ g ];` の構造。

**f = SET 発行済枚数 = 発行済枚数 + 1**

UPDATE文のSET句で、クーポン管理テーブルの発行済枚数をカウントアップする。更新後の発行済枚数がそのままクーポン発行連番になる。

**g = 発行上限枚数**

WHERE条件の `発行済枚数 < 発行上限枚数` で上限チェックを行い、上限に達していれば更新されない（以降の獲得ができない）。

**IPA公式：f = SET 発行済枚数 = 発行済枚数 + 1、g = 発行上限枚数**

---

### 設問4 正解：下線② = RU、下線③ = C

- **下線②（UPDATE処理）= RU（Read, Update）**：ロックあり方式では、クーポン管理テーブルに対して発行済枚数の読取り（R）と更新（U）が発生する。
- **下線③（INSERT処理）= C（Create）**：クーポン明細テーブルへのレコード追加（INSERT）はCreate（C）。

**IPA公式：下線②=RU、下線③=C**

## 参考：主要キーワード

| 用語 | 説明 |
|------|------|
| UNIQUE制約 | 指定列（または列の組み合わせ）に重複した値を持つ行の挿入・更新を禁止する制約 |
| 主キー制約（PRIMARY KEY） | テーブルの各行を一意に識別する列の制約。NULL不可・一意性保証 |
| ロックなし方式 | SELECT+INSERT で楽観的並行制御を行う方法。主キー重複時は再試行が必要 |
| ロックあり方式 | SELECT FOR UPDATE でロックを取得してから更新する悲観的並行制御 |
| COALESCE(A, B) | Aがnull以外ならA、nullならBを返す関数 |
| 排他制御 | 複数のトランザクションが同一データを同時更新しないための仕組み |
| CRUD図 | Create/Read/Update/Delete の操作をプロセスとデータの関係で表す図 |
| ALTER TABLE | 既存テーブルの定義を変更するSQL文（列追加・制約追加等） |
| 連番管理 | トランザクション環境でも重複や欠番なしに連番を発行する仕組み |
