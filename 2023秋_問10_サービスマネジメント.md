# 2023年秋期（令和5年度秋期）応用情報技術者試験 午後 問10（選択）
## サービスマネジメント：販売管理システム刷新に伴う SLA・サービスレベル管理（内部供給者との合意）

---

## 問題文

**問10** サービスレベルに関する次の記述を読んで、設問に答えよ。

E社は防犯カメラ・入退室認証機器・監視モニターなどのオフィス用セキュリティ機器を製造販売する中堅企業である。E社の販売部の販売担当者は、E社営業日の営業時間である9時から18時までの間、販売活動を行っている。E社の情報システム部は、販売管理システム（以下、現システムという）、製品管理システム、社内 Web システムなどを開発・運用し、社内の利用者にサービスを提供している。現システムは、納入先の所在地・納入先との取引履歴などの納入先情報を管理し、製品管理システムは24時間365日運用されており、E社の従業員は、業務に役立つ情報を、社内 Web システムを用いて、いつでも閲覧することができる。

情報システム部には、サービス課、システム開発課及びシステム運用課がある。サービス課は、システム運用及び IT 基盤の管理を担当する。

販売担当者は利用者として提供される販売管理サービス（以下、現サービスという）は現システムによって実現されている。例えば、納入先から製品の引き合いがあった場合、まず現システムで納入先情報を検索して、引き合いに関する情報を登録する作業に続いて、現システムと製品管理システムの両システムにアクセスして情報を検索したり、情報を登録したりする作業がある。最後に表示される納期と価格の提案書を要求する製品に関する情報を記録する（引き合いには当日の営業時間内に納入先に納期と価格の提案書を行うことを要求することもある）。引き合いが少ない場合は見積もり以上の要求もあることもあり、目標が達成できなくなる。販売部の行う納入先への提案は販売部の重要な事業機能なのであるので、販売部は現システムの改善を要求事項として情報システム部に提出していた。

そこで、情報システム部は、販売部の要求事項に対応するため、現システムに改修を加えたものを新システムとし、来年1月から新サービスとして提供することになった。

---

### 〔新サービスとサービスマネジメントの概要〕

販売部のG課長が業務要件を取りまとめ、システム開発課が現システムを改修し、システム運用課が IT 基盤を用いて新システムを運用し、販売担当当者が新サービスを現システムと同様に引き合いに情報を新システムに登録して、提案書情報作成を新システムに要求すると、新システムでは新システムと製品管理システムが連動して処理を実行し、提案に必要となる納期と価格の情報を表示する。

新サービスのサービスマネジメントについては、現サービス同様、サービス課販売サービスチームの F 君が担当する。サービス課は、従来からサービスデスク機能をコールセンター会社の Y 社に業務委託しており、外部からの問合せも、利用者からの問合せも、サービスデスクが窓口受付けとして対応し、問合せの内容が複雑な場合はエスカレーションされ、情報システム部が対応し、対応完了後、販売サービスチームはサービスデスクに連絡する。また、サービスデスクでは、サービスが利用できることを利用者に確認してサービス回復とする。システム運用課が障害箇所を特定する。その後、システム運用課が当該ストレージを復旧させ、販売サービスチームはサービスデスクに復旧の連絡をする。サービスデスクでは、サービスが利用できることを利用者に確認してサービス回復とする。サービスデスクが受け付けてからサービスが回復するまでの経過時間のことを、サービス回復時間と定める。

### 〔サービスレベル項目と目標の設定〕

社内に提供するサービスについて、情報システム部は、社内の利用部門との間で SLA を合意していなかったが、新サービスではサービスレベル項目と目標を明確にし、販売部と情報システム部との間で SLA を合意することにした。そこで、情報システム部の F 君が情報システム部長の指示の下で、販売部の要求事項と実現可能性を考慮しながらサービスレベル項目と目標の案を作成し、新サービスの利用開始に合わせて SLA のドラフトとして作成することとなった。F君が開発及び運用などのメンバーを含む全員、販売部の担当者も含む SLA のサービスレベル項目と目標を作成する責任者として F君が作成することとしてなった。

F君は、システム開発課がシステム設計を完了する前に、現システムで測定されているシステム評価指標を参考に、表1に示す販売部と情報システム部との間のサービスレベル項目と目標（案）を作成した。

### 〔サービス提供者とサービス供給者との合意〕

新サービスは、サービス課がサービス提供者となって、SLA に基づいて販売部にサービス提供をすることにした。サービスを提供するにあたって、外部供給者としてY社とシステム開発課及びシステム運用課が含まれる。

サービスデスクについては、従来、サービス課とY社との間で（以下、Y社への「合意」という）として文書化されている。このため、1か月前に Y社に件数を提案することになっている。Y社は、提供された追加件数に基づき合意件数の見積もりを行い、サービスデスク要員の体制を確保する。

F 君は、②**新サービスを契機として、サービス課と内部供給者との間で、サービスレベル項目と目標を合意することにした**。新サービスとシステム開発課との間の主要なサービスレベル項目と目標（案）を表2に示す。

> | 項番 | サービスレベル項目 | サービスレベル目標 |
> |---|---|---|
> | 1 | インシデントが発生した場合、システム課からインシデントの診断結果を提供する時間 | E社営業日の9時から20時まで |
> | 2 | 月次作業後に、システム開発課からサービス課へのインシデントの診断結果を提供する場合、システム開発課からサービス課へのインシデントの接続を受け付けてからシステムを復旧するまでの時間 | 8時間以内 |

### 〔受入テストにおける指摘と対応〕

システム開発課がシステム開発作業が完了し、新サービス開始の2週間前に販売部が参画する新サービスの受入れテストを開始した。受入れテストを行った結果、販売部から情報システム部に対して、次の評価と指摘が挙がった。

- 機能・性能なくなった問題はなく、新サービスを開始していいと判断できる。
- 新サービスの操作方法を説明したマニュアルは整備されているが、提案書情報作成を要求する処理においてサービスデスクへの問合せが必要になる点を要するような処理の説明が不足している点があった。

F君は、サービスデスクへの問合せ件数が想定よりも多くなる懸念を感じた。Y社担当者も調べたところ、④**新サービス開始の問合せ件数を削減するための対応が必要と考えた**。そこで、利用者が参照できる⑤**FAQ 社内 Web システムに掲載する**ことによって、新サービスの操作方法についてマニュアルで解決できなかった疑問が出た場合は、利用者自身で解決できるように準備をすることにした。

## 解答と解説

### 設問1

**正解：ウ（販売担当者が提案書情報作成を新システムに要求してから納期と価格の情報が表示されるまでに要する時間）**

G課長の要求事項は「引き合いの当日に納期と価格の提案書を作成する」こと。表1に抜けている性能指標は、新システムで提案書情報作成を要求してから応答が返るまでの処理時間である。

---

### 設問2

**(1) 正解：ウ（サービス要求事項に関する指示書）**

サービス課と Y 社の間で交わされる文書は「運営レベル合意書（OLA）」に相当するもので、サービスデスクへの要求事項を記述した指示書。

**(2) 正解：表1のサービスレベル目標の達成には、内部供給者との目標の合意が必要だから（38字）**

外部のサービスレベル（SLA）を達成するために、内部供給者（システム開発課・運用課）との間でも同等の目標（OLA）を合意する必要がある。

**(3) 正解：サービス回復時間にはシステム開発課以外で実施する作業の時間も含まれるから（38字）**

表2のサービスレベル目標（8時間以内でシステム復旧）と表1のサービスレベル目標（8時間以内のサービス回復）が同じ8時間では、サービスデスクへの連絡やユーザー確認の時間を含む余裕がない。

---

### 設問3

**(1) 正解：サービスデスク要員の体制が確保できないから（22字）**

Y社への問合せ件数は1か月前に提示するが、実際に想定より多くなると体制確保が間に合わない。問合せ件数を削減することで体制超過のリスクを低減する必要がある。

**(2) 正解：サービスデスクのサポート時間帯以外でも、利用者が疑問を解決できる。**

FAQ を社内 Web システムに掲載することで、サービスデスクの対応時間外（18時以降など）でも利用者が自己解決できるようになる。

## 参考：主要キーワード

| 用語 | 説明 |
|------|------|
| SLA（Service Level Agreement） | サービス提供者と利用者の間で合意するサービスレベルの文書 |
| OLA（Operational Level Agreement） | 組織内部の部門間で合意するサービスレベルの文書 |
| サービスレベル目標 | SLA で定める個々の指標の目標値 |
| サービス稼働率 | サービスが利用可能だった時間の割合（= 稼働時間 / 計画時間） |
| サービス回復時間 | 障害発生（受付）からサービス回復確認までの経過時間 |
| インシデント | サービスの停止・品質低下を引き起こす予期しない事象 |
| エスカレーション | 一次対応で解決できない問合せを上位の専門部門に引き継ぐプロセス |
| サービスデスク | 利用者からの問合せ・インシデントを一元受付するコンタクトポイント |
| FAQ | Frequently Asked Questions。よくある質問と回答集 |
| 内部供給者 | サービス提供者の内部でサービス実現を担う部門（システム開発課等） |
