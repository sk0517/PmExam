## 問題文

### 〔SQLの作成〕

T さんは、表1の項番2の業務要件を実現するための検索のSQL文を作成した。作成したSQL文を図2に示す。なお、「:カテゴリID」、「:下限価格」、「:上限価格」、「:商品状態」、「:出品状況」、「:キーワード」は、該当の値を格納する埋込み変数である。また、最上位であるカテゴリの上位カテゴリIDにはNULLが設定されている。

### 図2 作成したSQL文

![図2 SQL文](../images/2024秋_問06_図02_SQL文.png)

```sql
[　e　] 指定カテゴリ(カテゴリID, カテゴリ名, 上位カテゴリID) AS (
  SELECT A.カテゴリID, A.カテゴリ名, A.上位カテゴリID
    FROM カテゴリ A WHERE A.カテゴリID = :カテゴリID
  [　f　]
  SELECT B.カテゴリID, B.カテゴリ名, B.上位カテゴリID
    FROM カテゴリ B, 指定カテゴリ C WHERE B.上位カテゴリID = C.カテゴリID
)
SELECT * FROM 出品
  INNER JOIN 指定カテゴリ
  ON [　g　]
WHERE 出品.出品価格 BETWEEN :下限価格 AND :上限価格
  AND 出品.商品状態 = :商品状態
  AND 出品.出品状況 = :出品状況
  AND (出品.商品名 [　h　] OR 出品.商品説明 [　h　])
```

### 〔性能の検証と改善〕

T さんがテストデータを用いて図2のSQL文の実行性能を検証したところ、実行を開始してから検索結果が得られるまでの処理時間が長く、実用的ではないことが判明した。

本システムでは出品される商品の数が膨大であり、利用者が図2のSQL文を頻繁に実行することが予想される。そこで、T さんはキーワードでの検索が必要な商品名及び商品説明の列には全文検索エンジンを用いるとともに、その他の列に対しては適切なインデックスを設定し、性能上の懸念を解消することを検討した。

インデックスの方式には、B-treeインデックスを採用することにした。T さんは、各表の表定義を確認し、インデックスを設定すべき列を検討した。出品表の表定義を表2に、カテゴリ表の表定義を表3に示す。

表2及び表3のデータ型欄は、データ型、長さ、精度、位取りを示す。PK欄は主キー制約、UK欄はUNIQUE制約、非NULL欄は非NULL制約の指定をするかどうかを示す。指定する場合にはYを、指定しない場合にはNが記入されている。ここで、主キーに対してはUNIQUE制約は指定せず、非NULL制約は指定するものとする。カーディナリティ欄は列に多数の異なる値をもつ場合には高を、少数の異なる値をもつ場合には低を記入する。そして、高と低の中間の数の異なる値をもつ場合には中を記入する。データ分布欄は列に含まれる値の確率分布の仮定を示す。

T さんは、①<u>B-treeインデックスの特性</u>を踏まえて、特定の値を指定したときに行数を表全体の5%以下に絞り込める列だけにインデックスを設定することにした。

**解答群：**
- ア インデックスを設定した各列に対する条件をAND演算子で組み合わせた検索は高速化できるが、NOT演算子を用いた条件による検索は高速化できない。

## 解答と解説

### 設問1

**正解：**
- **a=カテゴリの自己参照の1対多関連（`→`）**：カテゴリが自分自身（上位カテゴリID→カテゴリID）を参照する1対多の関連。
- **b=取引と発送の1対1関連（実線）**：1つの取引に対して1件の発送が対応する1対1の関連。
- **c=追跡番号**：発送エンティティの属性。S社が管理し、配送業者のWebサイトと連携して配送状況を確認するために使う追跡番号。
- **d=利用者とフォローの1対多関連（`→`）**：利用者→フォロー（フォロー元・フォロー先）の1対多の関連。

**IPA公式：a=カテゴリの自己参照の1対多関連、b=取引─発送の1対1関連、c=追跡番号、d=利用者→フォローの1対多関連**

---

### 設問2

**正解：**
- **e=WITH RECURSIVE**：再帰的共通テーブル式（再帰CTE）を定義するキーワード。指定カテゴリを定義する先頭に置く。
- **f=UNION ALL**：再帰CTEの非再帰部（アンカー部）と再帰部を結合する演算子（重複も含めて全行）。
- **g=出品.カテゴリID = 指定カテゴリ.カテゴリID**：出品と指定カテゴリのJOIN条件。
- **h=LIKE '%' || :キーワード || '%'**：キーワードを部分一致で検索するパターン（商品名・商品説明の両方に指定）。

> 注：`[　e　]` はSQLの先頭（指定カテゴリの定義の前）、`[　f　]` はアンカー部と再帰部の間に位置する。

**再帰CTEの構造：**
```sql
WITH RECURSIVE 指定カテゴリ(カテゴリID, ...) AS (
  -- アンカー部（非再帰部）：指定カテゴリ自身
  SELECT ... FROM カテゴリ A WHERE A.カテゴリID = :カテゴリID
  UNION ALL
  -- 再帰部：上位カテゴリIDで自己結合してサブカテゴリを再帰的に取得
  SELECT ... FROM カテゴリ B, 指定カテゴリ C WHERE B.上位カテゴリID = C.カテゴリID
)
```

**IPA公式：e=WITH RECURSIVE、f=UNION ALL、g=出品.カテゴリID = 指定カテゴリ.カテゴリID、h=LIKE '%' || :キーワード || '%'**

### 設問3

**(1) 正解：ア、ウ、オ**

- **ア：正しい** - AND条件の組合せは高速化できるが、NOT演算子を用いた条件はインデックスで絞り込めない。
- **ウ：正しい** - 値の分布に偏りがない（一様分布）場合、検索性能が安定する。
- **オ：正しい** - B-treeでは特定行の検索の計算量はO(log n)。
- **イ：誤り** - 列に演算や型変換を行うとインデックスが使えず高速化できない。
- **エ：誤り** - カーディナリティが低い列はインデックスの効果が小さい。
- **カ：誤り** - 挿入の計算量もO(log n)（O(n)ではない）。
- **キ：誤り** - B-treeは範囲検索（BETWEEN等）も高速化できる。

**IPA公式：ア、ウ、オ**

**(2) 正解：**
- **出品表**：カテゴリID、出品価格
- **カテゴリ表**：上位カテゴリID

**理由：**
- `出品.カテゴリID`：INNER JOINの結合条件に使用される外部キー。カーディナリティ中で特定値により5%以下に絞り込める → B-treeインデックスが有効。
- `出品.出品価格`：BETWEEN句の範囲検索に使用。カーディナリティ高 → B-treeインデックスが有効。
- `カテゴリ.上位カテゴリID`：再帰CTEの結合条件 `WHERE B.上位カテゴリID = C.カテゴリID` に使用。カーディナリティ中 → B-treeインデックスが有効。
- 商品状態・出品状況はカーディナリティが低く（1〜3の3値、各値で約1/3）、特定の値で5%以下に絞り込めないためインデックス対象外。
- 出品ID・カテゴリIDの主キーには自動的にインデックスが設定されるため、明示的な追加は不要。

## 参考：主要キーワード

| 用語 | 説明 |
|------|------|
| 自己参照（再帰関連） | 同一エンティティの別インスタンスへの参照。カテゴリの階層構造などに使用 |
| WITH RECURSIVE（再帰CTE） | 自己参照するSQL文。階層構造のデータを再帰的に取得できる |
| UNION ALL | 集合演算子。重複行も含めてすべての行を結合（UNION は重複除去） |
| アンカー部/再帰部 | 再帰CTEの構成要素。アンカー部が初期行、再帰部が繰り返し参照する |
| B-treeインデックス | バランスド木構造のインデックス。完全一致・前方一致・範囲検索に有効 |
| カーディナリティ | 列の値のユニーク度。高カーディナリティほどインデックスの絞り込み効果が大きい |
| LIKE演算子 | 文字列のパターンマッチング。`%` は任意文字列、`_` は任意1文字 |
| 追跡番号 | S社が発送ごとに管理する番号。配送業者のWebサイトと連携して配送状況を確認 |
| 全文検索エンジン | 大量テキストの高速検索に特化したエンジン。B-treeより部分一致検索が高速 |
| 共通テーブル式（CTE） | WITH句で定義される一時的な名前付き結果セット。クエリを読みやすくする |
