# 2023年春期（令和5年度春期）応用情報技術者試験 午後 問9（選択）
## プロジェクトマネジメント：金融機関クラウド移行プロジェクト

---

## 問題文

**問9** 金融機関システムの移行プロジェクトに関する次の記述を読んで、設問に答えよ。

P社は、本店と全国38か所の支店（以下、拠点という）から成る国内の金融機関である。P社では、土日祝及び年末年始を除いた日（以下、営業日という）に営業をしている。P社では、金融商品の販売業務を行うためのシステム（以下、販売支援システムという）をオンプレミスで運用している。

販売支援システムは、営業日だけ稼働しており、拠点の営業員及び拠点を統括する訪問者の単位で販売実績の管理を行っている。販売支援システムの運用・保守及びサービスデスクは、情報システム運用課（以下、運用課という）が担当し、サービスデスクが解決できない場合のエスカレーション対応及びシステム関係は、情報システム開発課（以下、開発課という）が担当する。

販売支援システムのハードウェアは、P社内に設置されたサーバ機器、拠点の端末、及びサーバと端末を接続するネットワーク機器で構成される。

販売支援システムのアプリケーションソフトウェアのうち、中心となる機能は、X社のクラウドシステム（以下、Xパッケージという）を利用しており、一部の標準機能が不足する機能は、Xパッケージをカスタマイズしている。

販売支援システムのサーバ機器及びXパッケージはいずれも来年3月末に保守契約の期限を迎え、いずれも老朽化しているため以後保守費用が大幅に上昇する。そこで、P社は、本年4月に、クラウドサービスを活用して現状のサーバ機器を撤去した上でとる構築期間とコストの削減策を実現し、さらにXパッケージをバージョンアップして大幅な機能改善を図ることを目的に移行プロジェクトを立ち上げた。X社から、今回適用するパージョンは、OS やミドルウェアに制約があると報告された。

開発課のQ課長を、移行プロジェクトのプロジェクトマネージャ（PM）に任命し、移行プロジェクトの計画の作成に着手した。Q課長は、開発課のR主任に、現在の販売支援システムからの移行作業、同課のS主任に移行先のクラウドサービスでのシステム構成、移行作業とそのスケジュールの調整などを指示した。

### 〔ステークホルダの要求〕

Q課長は、まず、ステークホルダマネジメントについて検討した。Q課長は経営層、商品販売部及び情報システム部が参加するステアリングコミッティを設置し、移行プロジェクトの進捗状況の報告、重要なリスク及び対応方針の報告、最終的移行判定などを行うことにした。

Q課長は、移行方式について、全拠点で一斉に切り替える①**一括切替方式**を採用したとした。そこで、Q課長は、商品販売部に、サービスデスクから受けるエスカレーション対応のリソースを拡充することで、移行直後に発生する問合せに迅速に回答することを説明して了承を得た。

現行の販売支援システムのサーバ機器及びXパッケージの保守期限が来年3月末であるため、来年3月末までに移行を完了する必要がある。Q課長は、移行を期に期限前に切替作業に問題が生じた場合に備えて、年末年始に切替作業を行うことにした。

---

Q課長は、移行の目的や制約を検討した結果、IaaS型のクラウドサービスを採用することにした。IaaSベンダーの選定に当たり、S主任は、新システムのセキュリティインシデントの発生に備えて、セキュリティ対策をP社セキュリティポリシーに基づいて策定するように定めるとした。Q課長は、②**具体的なセキュリティ対策に先立って実施すべきことを**S主任に指摘した。S主任は、Q課長の指摘を踏まえて作業を進め、セキュリティ対策を策定した。

最後に、Q課長は、これまでの検討結果をまとめに、IaaSベンダーの③**RFPを提示**し、受領した提案内容を評価した。その評価結果に基づき、M社を選定した。

Q課長は、これについて経営層に報告し承認の承認を受けた。

### 〔移行プロジェクトの作業計画〕

R主任とS主任は、移行手順の作成、移行ツールの開発、移行統合テスト、営業員の教育・訓練及び受入れテスト、移行リハーサル、本番移行、及びその後の初期サポートについて作業計画を作成した。主な作業の内容は次のとおりである。

**(1)** 移行手順書の作成

移行に関わる全作業の手順書を作成し、開発するメンバーでレビューする。

**(2)** 移行ツールの開発

移行作業の実施に当たって、データ変換ツール、構成管理ツールなどのX社提供のツールを活用する。Xパッケージをカスタマイズしていたため、X社提供のデータ変換ツールを利用することができないので、移行に必要なデータ変換機能を開発課が追加開発することになった。

**(3)** 移行統合テスト

移行統合テストでは、移行ツールが正常に動作し、移行手順書どおりに作業できることを確認する。移行後のシステムの全てのことを全部テストすることを意図している。移行統合テストで本番データを検証するため、X社テスト規定で、個人情報を含んだ本番データをテスト目的で利用しないこと、と定められている。P社テスト規定でも、個人情報を含んだ本番データをテスト目的で利用しないこと、と定められている。本番データを使用する場合には、その必要性を明らかにした上で、個人情報を個人情報保護法及び関連ガイドラインに従って匿名加工情報に加工する処置をして用いること、と定められている。そこで、R主任は本番データに含まれる個人情報を匿名加工情報に加工して使用する計画を作成した。Q課長はその計画を承認した。その際、R主任は、④**ある手続**を実施した上で対応方針を決定した。

**(4)** 営業員の教育・訓練及び受入れテスト

商品販売部の部員が、S主任及び拠点長の責任者のもとで教育・訓練のスケジュールを組み立てた。これに従って、営業員の業務終了後に受入れテストを実施し、商品販売部の部員及び全営業員に対する教育・訓練を実施する。

**(5)** 移行リハーサル

移行リハーサルでは、移行統合テストで検証した移行ツールを使用した移行手順、本番移行の当日の作業、及びタイムチャートを検証する。

**(6)** 本番移行

移行リハーサルで検証した一連の手順に従って切替作業を実施する。本番移行は本年12月31日〜来年1月2日に実施することに決定した。

**(7)** 移行後の初期サポート

移行後のトラブルや問合せに対応するための初期サポートを実施する。初期サポートの実施に当たり、Q課長は、移行後も、システムが安定稼働して問い合わせがサービスデスクに収束するまでの間、⑤**ある支援を継続**するよう S主任に指示した。

Q課長は、このの検討結果をまとめて、⑥**新システムの移行可否を評価するうえで必要な文書**の作成に着手した。

## 解答と解説

### 設問1

**(1) 正解：営業日に業務の停止が不要（13字）**

一括切替方式では、年末年始など非営業日にシステム切替を行う。全拠点を同時に切り替えることで、段階移行と比較して業務停止日数が少なく、営業日に業務停止を強いる必要がない。商品販売部（営業側）にとってのメリット。

**(2) 正解：エ（セキュリティ対策の責任範囲の明確化）**

IaaSを利用する場合、P社とクラウドベンダー（M社）間でセキュリティ対策の責任範囲（責任共有モデル）を明確化しておかなければ、具体的な対策の立案ができない。先に責任範囲を明確にすることが必要。

**(3) 正解：IaaS利用による構築期間とコスト（17字）**

RFPでは、プロジェクトの目的である「クラウドサービスを活用した構築期間とコストの削減」を実現できるかを重視して評価する。

---

### 設問2

**(1) 正解：ステアリングコミッティで本番データを用いたテストの承認を得る。（31字）**

P社テスト規定で本番データ利用時は「必要性の明示と匿名加工処置」が必要。また、個人情報を含む本番データの利用は重要な意思決定事項のため、ステアリングコミッティでの承認手続が必要となる。

**(2) 正解：エスカレーション対応の開発課リソースの拡充（22字）**

移行後の初期サポート期間中、問合せがサービスデスクに収束するまでは、通常を超える問合せ・トラブルが発生する可能性がある。そのため、サービスデスクのエスカレーション対応（開発課）のリソースを継続的に拡充しておく。

**(3) 正解：移行判定基準書（8字）**

新システムへの移行可否を評価するための基準・判定条件をまとめた文書。ステアリングコミッティでの最終判定に使用する。

---

### 設問3

**(1) 正解：a = オ（発生確率・影響度マトリックス）、b = ア（感度分析）**

| 空欄 | 正解 | 解説 |
|------|------|------|
| **a** | オ（発生確率・影響度マトリックス） | リスクの定性的分析に使用。発生確率と影響度の組み合わせでリスクの優先度を視覚化する |
| **b** | ア（感度分析） | 移行作業に最も影響が大きいリスクを特定する定量的分析手法。リスクの相対的な重大性を評価 |

**(2) 正解：追加発生する保守費用の確保（14字）**

来年3月末に現行システムの保守契約が切れるため、移行が遅延した場合は保守期間を延長する必要がある。その際に追加で発生するサーバ機器・Xパッケージの保守費用を確保することが対応策となる。

## 参考：主要キーワード

| 用語 | 説明 |
|------|------|
| 一括切替方式 | 全拠点・全システムを同時に新システムへ切り替える移行方式 |
| 段階移行方式 | 一部拠点・機能から段階的に切り替える移行方式 |
| ステアリングコミッティ | プロジェクトの重要決定を行う経営層・関係部門で構成された委員会 |
| IaaS（Infrastructure as a Service） | サーバ・ネットワーク等のインフラをクラウドで提供するサービス |
| RFP（Request for Proposal） | ベンダーに提案を求めるための提案依頼書 |
| 責任共有モデル | クラウドベンダーと利用者間でセキュリティ対策の責任範囲を分担する考え方 |
| 移行判定基準書 | 本番移行の可否を判断するための基準・条件をまとめた文書 |
| 発生確率・影響度マトリックス | リスクの発生確率と影響度を評価して優先度を分類するマトリックス図 |
| 感度分析 | リスク要因の変化がプロジェクト全体に与える影響の大きさを分析する手法 |
| 匿名加工情報 | 個人情報を特定できないよう加工した情報。テスト目的での利用に使用 |
